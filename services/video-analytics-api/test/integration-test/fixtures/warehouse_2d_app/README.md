# warehouse_2d_app fixtures

Fixtures for integration tests covering **warehouse_2d_app** with data from the Elasticsearch dump (time range **2026-02-14T10:13:51.292Z** to **2026-02-14T10:17:33.492Z**, place `building=Warehouse/room=Room-1`).

- **mdx-behavior**: GET /behavior — input/output fixtures per test case (sensorId, place, timestamps, objectId, objectType, queryString, maxResultSize combinations).
- **mdx-frames**: GET /frames — sensorId + frameId or sensorId + fromTimestamp/toTimestamp.
- **mdx-incidents**: GET /incidents — sensorId or place, with optional timestamps and maxResultSize.
- **mdx-events**: GET /events/tripwire and GET /events/roi — sensorId or place + timestamps.

**mdx-raw**: There is no GET API for raw frame data. The Elasticsearch dump contains `mdx-raw-*` indices; that data is ingested and exposed indirectly via GET /frames (mdx-frames) and other endpoints. No fixtures are generated for mdx-raw.

Generated by (from repo root): `node test/integration-test/scripts/run_warehouse_2d_app_tests.js <BASE_URL> [WEB_APIS_ROOT] [FIXTURES_DIR]`  
Or from `test/integration-test`: `node scripts/run_warehouse_2d_app_tests.js <BASE_URL> [WEB_APIS_ROOT] [FIXTURES_DIR]`

Each subdirectory (e.g. `behavior_sensorId_Camera`) contains:
- `input.json` — query parameters used for the request (validated with OpenAPI/AJV).
- `output.json` — response body (validated with OpenAPI spec).

**Why some outputs have empty arrays:** Fixtures like `{"tripwireEvents": []}`, `{"roiEvents": []}`, or `{"behaviors": []}` are valid API responses. The API returns 200 with an empty array when no documents match the query (time range, place, sensorId, objectType, etc.). The dump may not contain tripwire/roi events in the index or format the events API queries, or a filter (e.g. `place` + `objectType=Pallet`) may match no behaviors. The fixture still records the real request/response for that combination.
